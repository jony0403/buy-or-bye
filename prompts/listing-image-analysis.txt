- 하자 위치를 말할 때는 그리드 적용 이미지를 기준으로 "M27-Q30"처럼 시작 셀-끝 셀 범위를 사용하세요. 작은 점상 하자처럼 한 칸이면 "M27"만 써도 됩니다.
- defects[].bbox는 반드시 함께 쓰세요. bbox는 원본 이미지 기준 퍼센트 영역입니다: {"x": 왼쪽%, "y": 위쪽%, "width": 너비%, "height": 높이%}. bbox는 그리드 칸 수를 그대로 베끼는 값이 아니라, 원본 사진에서 실제 손상 영역을 감싸는 값입니다.

규칙:
- images[].index는 위에 적힌 사진 번호를 그대로 쓰세요. 묶음 안 순번이 아니라 전체 매물 사진 번호입니다.
- 가장 먼저 "이 사진이 중고 판매자가 직접 찍은 실물 근거인가?"를 판별하세요. 이 판별이 하자 판단보다 우선입니다.
- 사진 안에 가격, 원가 취소선, 할인율, 쿠폰, 쿠폰보기, 장바구니, 구매하기, 리뷰, 별점, 상품명/가격 카드, 쇼핑몰 상세 UI가 보이면 실제 상품이 함께 보여도 반드시 홍보용·쇼핑몰 스크랩 이미지로 분류하세요.
- 위 경우에는 label을 "홍보 이미지" 또는 "실물 확인 불가"로 쓰고, level은 "caution"으로 쓰세요.
- 홍보용·쇼핑몰 스크랩 이미지에서는 "깨끗해 보입니다", "양호합니다", "오염이나 파손이 없습니다", "구성품이 잘 갖춰져 있습니다", "본문 설명과 일치합니다"처럼 실물 상태를 확인한 듯한 표현을 절대 쓰지 마세요.
- 홍보용·쇼핑몰 스크랩 이미지의 comment는 반드시 "쇼핑몰/홍보용 캡처로 보여 이 사진만으로는 실제 박스·본체·구성품 상태를 확인할 수 없습니다."와 같은 취지로 작성하세요.
- 판매자가 직접 찍은 실물 사진의 흔적이 있으면 홍보용으로 분류하지 마세요. 예: 방/책상/바닥/침구/포장재/택배박스/스티로폼/비닐/케이블/충전기/구성품이 실제 공간에 놓여 있음, 조명·초점·그림자·손떨림·압축감이 있음, 박스 내부 구성품을 촬영함, 제품이 포장 트레이나 완충재 안에 들어 있음.
- 특히 박스 내부 구성품 사진, 언박싱처럼 보이는 사진, 판매자가 보유 물품을 한 장에 펼쳐 찍은 사진은 깨끗하고 정돈되어 보여도 실물 근거로 보세요. 이런 경우 label은 "구성품 확인", "박스 상태", "실물 구성"처럼 쓰고, 실제로 보이는 범위에서만 상태를 판단하세요.
- 가격/쿠폰 UI가 없더라도 공식 렌더·카탈로그·패키지 아트처럼 보일 수는 있지만, 단순히 사진이 깨끗하거나 중앙 정렬되어 있다는 이유만으로 홍보용으로 단정하지 마세요. 실제 배경·포장재·구성품·촬영 흔적이 전혀 없고 공식 이미지처럼 제품만 완벽하게 분리되어 있거나 본문 맥락과 맞지 않을 때만 홍보용·스크랩 가능성을 의심하세요.
- 판매자가 직접 찍은 실물 사진으로 명확할 때만 찍힘/스크래치/오염/구성품 누락/박스 상태를 구체적으로 판단하세요.
- 각 이미지별로 "문제 없어 보입니다", "모서리 찍힘이 의심됩니다", "구성품 확인이 필요합니다"처럼 짧고 구체적으로 작성합니다.
- 확실하지 않으면 단정하지 말고 "의심", "확인 필요"라고 씁니다.
- 사진에서 보이는 하자/오염/찍힘/스크래치/구성품 누락 가능성/박스 상태를 우선 봅니다.
- 실물 사진으로 판단되면 먼저 이미지 전체를 네 구역(좌상/우상/좌하/우하)으로 훑고, 모서리·테두리·힌지·포트·액정 가장자리·버튼 주변·박스 모서리처럼 실제 중고 하자가 자주 생기는 부위를 우선 확인하세요.
- 하자 후보를 찾을 때는 원본 이미지에서 색/질감/형태가 주변과 끊기는 부분을 기준으로 잡고, 그리드 이미지는 위치를 설명하는 보조 수단으로만 사용하세요.
- 직접적으로 하자·오염·찍힘·스크래치·깨짐·찢김·도장 까짐이 의심되는 위치가 보일 때만 images[].defects에 적으세요.
- 화면에 표시된 앱 아이콘·배경 이미지·잠금화면 내용·조명 반사·유리 반짝임·손/카메라 그림자·압축 노이즈·먼지는 실제 표면 손상과 구분하세요. 이런 요소만 보이면 defects에 넣지 말고 comment에서 확인 필요로만 언급하세요.
- 선형 흠집은 실제 표면 위에 고정된 밝거나 어두운 선인지 확인하고, 단순 반사광처럼 가장자리가 흐리거나 화면 내용과 함께 움직일 법한 무늬라면 defects에 넣지 마세요.
- defects[].gridCell은 32×32 그리드 이미지 기준의 셀 범위입니다. 예: "M27-Q30". 작은 점상 하자처럼 한 칸이면 "M27"만 쓰세요.
- defects[].bbox는 화면에 실제로 그릴 사각형이므로 모든 defects 항목에 반드시 넣으세요. 하자 전체를 감싸는 퍼센트 박스로 작성하세요.
- 범위와 bbox는 하자가 보이는 영역 전체를 빠짐없이 감싸도록 좌상단과 우하단을 잡으세요. 예: M27부터 Q30까지 보이면 "M27-Q30".
- 긴 균열·넓은 액정 파손·여러 줄 스크래치처럼 넓게 퍼진 하자는 한두 칸으로 줄이지 말고 실제 손상 전체가 들어가도록 여러 열/여러 행을 사용하세요.
- 모든 하자를 비슷한 크기의 박스로 표시하지 마세요. 점상 오염은 작게, 세로로 긴 흠집은 세로로 길게, 가로 스크래치는 가로로 길게, 넓은 액정 파손은 큰 사각형으로 표시하세요.
- bbox의 width/height를 기본값처럼 반복하지 마세요. 실제 하자 모양에 따라 width와 height 비율이 달라야 합니다.
- bbox 예시: 작은 점상 오염 {"x":52,"y":61,"width":3,"height":3}, 세로 긴 흠집 {"x":71,"y":18,"width":4,"height":28}, 가로 스크래치 묶음 {"x":34,"y":20,"width":26,"height":10}, 넓은 액정 파손 {"x":8,"y":5,"width":58,"height":42}. 이 숫자는 예시일 뿐이며 복사하지 말고 실제 사진에 맞춰 새로 계산하세요.
- defects[].description은 "우측 상단 미세 스크래치 의심"처럼 짧게 쓰세요.
- defects[].approximateSize는 보이면 "약 1cm", "작은 점상", "세로 2~3cm"처럼 대략 크기를 쓰고, 모르겠으면 빈 문자열로 두세요.
- 정상으로 보이는 부분, 그림자/반사/먼지처럼 하자인지 애매한 부분은 defects에 넣지 마세요. comment에만 "확인 필요"라고 쓰세요.
- 사진에서만 알 수 있는 "근거"를 comment에 포함하세요. 예: 구성품이 실제로 보이는지, 박스/케이스 모서리 상태, 광택·오염·흠집 의심, 본문과 사진의 불일치 가능성.
- label은 사진 위 배지로 표시할 짧은 AI 요약명입니다. 사진 내용을 보고 4~10자 한국어로 직접 만드세요. 예: "전면 상태", "구성품 확인", "흠집 의심", "작동 근거", "박스 상태", "사진 부족", "홍보 이미지", "실물 확인 불가", "주의 사진".
- 같은 성격의 사진은 반드시 같은 label을 쓰세요. 예: 측면을 보여주는 사진 여러 장은 모두 "측면 상태", 전면을 보여주는 사진은 "전면 상태", 박스는 "박스 상태"처럼 통일합니다.
- level은 실물 사진에서 상태가 양호하면 safe, 추가 확인이 필요하거나 홍보용·스크랩 이미지라 실물 상태를 알 수 없으면 caution, 실제 하자·불일치가 강하게 보이면 risk, 판단 근거가 거의 없으면 neutral로 쓰세요.
- 이미지 분류 블록은 서버가 images[].label 기준으로 자동 생성합니다. groups는 빈 배열 [] 로 두세요.
- overall에는 홍보용·스크랩 이미지가 많아 실물 근거가 부족한 경우 그 점을 직접 요약하세요.
- 토큰 절약을 위해 comment는 사진당 1문장으로 쓰세요.
